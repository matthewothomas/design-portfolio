Bupa 2022–Present Token Architecture

Token System Scaling

Architecting a comprehensive token system from 30 app-specific values to 320+ cross-platform primitives and semantic tokens.

Token Architecture
Overview

The Challenge

Bupa's design tokens were born in the mobile app. They worked well for that context, but as the design system expanded to web and email, the limitations became clear. Hard-coded values were scattered across platforms with no shared language between designers and developers.

Designers were making decisions in isolation. A colour used for primary actions in the app might be completely different on web. Spacing was inconsistent. And when brand updates came, there was no single place to make changes.

The Problem

30 Tokens Built for One Platform

The original token set was tightly coupled to the app:

  • 30 tokens — mostly colour values, some spacing
  • No hierarchy — every token was a single value
  • Platform-specific naming — tokens referenced iOS patterns
  • No semantic meaning — colours were named by their value, not their purpose
  • Web and email used raw values — no tokens at all

Approach

Primitives + Semantic Architecture

The solution was a two-tier token architecture that separated raw values from their intended use:

Primitive Tokens (~150+)

The raw building blocks: colour palettes, spacing scales, typography values, and border properties. Named by their properties, not their usage.

Contextual tokens that describe intent, not values. Organised by category with full state coverage: default, hover, active, disabled, and focus states.

Semantic Typography Tokens (~40+)

Composite tokens combining font-family, size, weight, line-height, letter-spacing, and paragraph-spacing into reusable type styles.

Cross-Platform

One System, Three Platforms

The token architecture serves web, app, and email from the same source with platform-specific font families:

  • Web — Montserrat (brand font), CSS custom properties
  • iOS/Android — SF Pro / Roboto (native fonts), React Native implementation
  • Email — Arial (email-safe fallback), compiled to inline styles

When a token value changes, it propagates across all platforms. No more hunting through codebases to update colours. No more platform drift.

Outcome

Consistent Design Decisions at Scale

The scaled token system fundamentally changed how design decisions flow through the organisation:

  • 320+ tokens serving all platforms from a single source
  • Semantic naming — designers and developers speak the same language
  • Full state coverage — hover, active, disabled, focus states built in
  • Theme-ready architecture — dark mode and brand variations now possible
  • Faster brand updates — change once, propagate everywhere
  • Reduced errors — no more hard-coded values slipping into designs
